# Tomorrow TODO - Vue Memory Optimization Guide

## Goal

사내 환경에서 실행 중인 Vue 프로젝트의 메모리 사용량이 약 4GB 수준으로 관측되었을 때,
원인을 구분하고 우선순위 높은 최적화 작업을 안전하게 적용할 수 있도록 상세 가이드를 제공한다.

이 문서는 다음을 목표로 한다.

- 메모리 사용량이 실제 문제인지 먼저 판별
- Node(Vite), 브라우저 탭, DevTools, 사내 보안 모듈 중 어디가 원인인지 구분
- 현재 코드베이스에서 메모리 증가 가능성이 높은 구조를 식별
- 사내 개발자가 바로 수정할 수 있도록 파일 단위 수정 가이드 제공
- 수정 후 검증 기준과 체크리스트 제공

---

## 1. First: 4GB가 무엇의 메모리인지 구분

최적화 전에 반드시 아래를 먼저 확인한다.

### 1.1 확인 대상

- `node.exe` 또는 `vite` 프로세스 메모리
- 브라우저 탭(Chrome/Edge renderer) 메모리
- DevTools 메모리
- 사내 보안 모듈/확장 프로그램 메모리

### 1.2 권장 확인 방법

1. Windows 작업 관리자 열기
2. `node.exe` 메모리 확인
3. 브라우저 작업 관리자 열기
   - Chrome: `Shift + Esc`
   - Edge: 브라우저 작업 관리자 메뉴
4. 프로젝트 탭의 메모리 확인
5. DevTools를 켠 상태/끈 상태 각각 비교
6. 홈 / 대시보드 / 채팅 페이지를 각각 들어가며 증가 폭 비교

### 1.3 해석 기준

- `node.exe`가 비정상적으로 큰 경우:
  - Vite dev server, HMR, sourcemap, 플러그인 영향 가능성
- 브라우저 탭이 비정상적으로 큰 경우:
  - DOM 렌더링, 대량 데이터, 잦은 reactive 계산, 빈번한 repaint 가능성
- DevTools를 켰을 때만 급격히 커지는 경우:
  - 개발 환경 특성일 수 있으며 운영 영향과 분리해서 판단

## 2. 현재 코드베이스에서 우선 의심할 지점

### 2.1 App 전체 페이지 eager import

파일:

- `src/App.vue`

현재 구조:

- 주요 페이지 컴포넌트를 모두 정적 import 하고 있음
- 사용자가 홈 화면만 보고 있어도 다른 페이지 코드가 모두 메모리에 올라감

문제 포인트:

- dev 환경에서 HMR + sourcemap과 결합되면 초기 메모리 점유를 키움
- 큰 컴포넌트가 많을수록 효과가 커짐

현재 관련 import:

- `ChatbotPage.vue`
- `DashboardSection.vue`
- `FeaturesSection.vue`
- `LogDownloadPage.vue`
- `HowItWorksSection.vue`
- `UseCasesSection.vue`
- `TestimonialsSection.vue`
- `SecuritySection.vue`
- `CTASection.vue`

### 2.2 대시보드 컴포넌트 내부의 정적 대량 데이터

파일:

- `src/components/DashboardSection.vue`

특징:

- 필터 옵션, 메뉴 정의, 차트 샘플, 테이블 컬럼, 샘플 row 데이터가 한 파일 안에 집중되어 있음
- `baseRequests`, `generatedRequests`, `ceRequests`가 컴포넌트 내부에 존재
- `filteredRequests`, `sortedRequests`, `paginatedRequests`가 연쇄 computed 구조

문제 포인트:

- 데이터가 커지면 필터/정렬마다 새 배열이 계속 생성됨
- 현재는 샘플 데이터가 작지만, 실데이터 확장 시 브라우저 메모리와 GC 부담 증가 가능

### 2.3 사이드바 내부의 많은 reactive mock state

파일:

- `src/components/TheSidebar.vue`

특징:

- 권한/기능/기타 설정, 대화 이력, 계정 정보, role mapping 등이 한 파일에 모여 있음
- 일부 값은 바뀔 필요가 거의 없는데 `ref` 기반 reactive 상태로 선언되어 있음

문제 포인트:

- 단일 파일 내부에서 반응성 추적 대상이 많아짐
- 절대적인 메모리 사용량보다는 불필요한 reactive cost 증가

### 2.4 채팅의 글자 단위 스트리밍 업데이트

파일:

- `src/components/ChatbotPage.vue`

특징:

- `setInterval`로 문자열을 한 글자씩 append
- 거의 매 tick마다 `scrollToBottom()` 호출

문제 포인트:

- 메모리 누수보다는 렌더링/레이아웃 비용이 큼
- 긴 응답 테스트 시 브라우저 렌더러 부담 증가 가능

## 3. 최적화 우선순위

### 3.1 1순위 - 페이지 단위 lazy loading

가장 먼저 적용할 것.

대상 파일:

- `src/App.vue`

목표:

- 사용자가 현재 보고 있는 페이지 코드만 로드
- 홈만 띄웠을 때 대시보드/채팅/설정 페이지 코드까지 한 번에 올리지 않도록 변경

권장 방식:

```js
import { computed, defineAsyncComponent, defineComponent, h, ref } from 'vue'

const ChatbotPage = defineAsyncComponent(() => import('./components/ChatbotPage.vue'))
const DashboardSection = defineAsyncComponent(() => import('./components/DashboardSection.vue'))
const FeaturesSection = defineAsyncComponent(() => import('./components/FeaturesSection.vue'))
const LogDownloadPage = defineAsyncComponent(() => import('./components/LogDownloadPage.vue'))
const HowItWorksSection = defineAsyncComponent(() => import('./components/HowItWorksSection.vue'))
const UseCasesSection = defineAsyncComponent(() => import('./components/UseCasesSection.vue'))
const TestimonialsSection = defineAsyncComponent(() => import('./components/TestimonialsSection.vue'))
const SecuritySection = defineAsyncComponent(() => import('./components/SecuritySection.vue'))
const CTASection = defineAsyncComponent(() => import('./components/CTASection.vue'))
```

주의:

- `TheSidebar.vue`는 항상 보이는 영역이라 유지 가능
- `HomePage` 내부에서도 필요 시 `HeroSection`, `LogoStrip`, `TheFooter`를 lazy loading할 수 있으나 1차 단계에서는 페이지 단위만 적용해도 충분

### 3.2 2순위 - 큰 정적 데이터 분리

대상 파일:

- `src/components/DashboardSection.vue`
- `src/components/TheSidebar.vue`
- `src/components/HowItWorksSection.vue`
- `src/components/LogDownloadPage.vue`

목표:

- 컴포넌트 파일에서 큰 상수/더미 데이터 제거
- 정적 정의를 `src/data/` 또는 `src/mocks/` 아래로 분리

권장 예시 폴더:

```text
src/data/dashboardData.js
src/data/sidebarData.js
src/data/logDownloadData.js
src/data/howItWorksData.js
```

분리 대상 예시:

- 메뉴 배열
- 필터 옵션
- 차트 샘플
- 샘플 요청 목록
- 대화 이력 mock
- 권한/설정 옵션

장점:

- 컴포넌트 가독성 개선
- 메모리 측정보다 유지보수성 개선 효과가 먼저 큼
- 필요 시 동적 import로 추가 최적화 가능

### 3.3 3순위 - 불필요한 reactive state 축소

대상 파일:

- `src/components/TheSidebar.vue`
- `src/components/DashboardSection.vue`

원칙:

- 사용자가 수정하지 않는 값은 `ref`, `reactive` 대신 일반 `const` 사용
- 불변 데이터는 `Object.freeze()` 고려

예시:

- `settingsTabs`
- `accountInfo`
- `navItems`
- `dashboardSubItems`
- `roleOptions`
- `filterControls`
- `tableColumns`

유지해야 하는 reactive 예시:

- 사용자가 직접 변경하는 설정 토글
- 열린/닫힌 상태
- 선택된 필터
- 현재 페이지 번호

### 3.4 4순위 - 채팅 스트리밍 배치 처리

대상 파일:

- `src/components/ChatbotPage.vue`

현재 문제:

- 한 글자씩 렌더링
- 매 글자마다 스크롤 이동

개선 원칙:

- 문자열 append를 글자 단위가 아닌 chunk 단위로 변경
- `scrollToBottom()` 호출을 throttle 또는 `requestAnimationFrame` 기준으로 제한

권장 방향:

```js
const chunkSize = 8
const streamDelay = 35
```

또는:

- 일정 간격마다 여러 글자를 붙임
- 1 frame에 1회만 scroll 수행

효과:

- 긴 응답에서 렌더링 부하 감소
- UX는 거의 유지하면서 브라우저 부담 완화

### 3.5 5순위 - 대시보드 대량 데이터 대비

대상 파일:

- `src/components/DashboardSection.vue`

실데이터가 커질 경우 적용 검토:

- 서버 측 필터링/정렬/페이지네이션 유지
- 한 번에 수백~수천 행을 렌더링하지 않기
- 필요 시 virtualization 도입

후보:

- `vue-virtual-scroller`

단, 현재 사내 반영 1차에서는 lazy loading과 데이터 분리가 먼저다.

## 4. 실제 수정 지시서

### 작업 1. App.vue를 페이지 lazy loading 구조로 변경

파일:

- `src/App.vue`

해야 할 일:

1. 각 페이지 컴포넌트의 정적 import 제거
2. `defineAsyncComponent` 기반으로 전환
3. 기존 `pageMap` 유지
4. 라우팅 동작이 깨지지 않는지 확인

검증:

- 홈 진입 정상
- 사이드바 메뉴 이동 정상
- 채팅 진입 정상
- 대시보드 진입 정상

### 작업 2. DashboardSection의 정적 데이터 분리

파일:

- `src/components/DashboardSection.vue`

새 파일 예시:

- `src/data/dashboardData.js`

분리 후보:

- `filterControls`
- `dashboardMenus`
- `teamDistribution`
- `teamWorkload`
- `tableColumns`
- `ceFilterControls`
- `ceTableColumns`
- `baseRequests`
- `requestVariants`
- `statuses`
- `tcVersions`
- `assignees`
- `ceRequests`

주의:

- `computed`, `watch`, `ref`, `reactive`와 직접 연결된 상태만 컴포넌트에 남김
- 데이터 분리 후 import 경로만 교체

### 작업 3. TheSidebar의 정적/동적 상태 분리

파일:

- `src/components/TheSidebar.vue`

새 파일 예시:

- `src/data/sidebarData.js`

분리 후보:

- `settingsTabs`
- `accountInfo`
- `navItems`
- `dashboardSubItems`
- `roleOptions`

검토 후보:

- `chatHistories`
- `roleAccounts`

주의:

- 실제 UI에서 수정 가능한 항목만 `ref` 유지
- 단순 표시용 정적 데이터는 일반 상수 유지

### 작업 4. ChatbotPage 스트리밍 비용 완화

파일:

- `src/components/ChatbotPage.vue`

해야 할 일:

1. 한 글자씩 append하는 로직을 chunk append로 변경
2. `scrollToBottom()`를 tick마다 부르지 않도록 제한
3. 긴 응답에서도 UI 지연이 없는지 확인

예시 방향:

```js
const chunkSize = reply.length > 500 ? 12 : 6
assistantMsg.content += reply.slice(i, i + chunkSize)
i += chunkSize
```

그리고:

- 일정 횟수마다만 `scrollToBottom()`
- 또는 `requestAnimationFrame` 내부에서만 scroll

## 5. 수정 후 측정 절차

### 5.1 동일 조건 비교

최적화 전후 모두 같은 조건에서 측정한다.

- 같은 브라우저
- 같은 탭 수
- 같은 DevTools 상태
- 같은 시작 페이지

### 5.2 체크 포인트

1. 앱 첫 진입 직후 메모리
2. 홈 화면 1분 유지 후 메모리
3. 대시보드 진입 후 메모리
4. 채팅 긴 응답 스트리밍 후 메모리
5. 페이지를 여러 번 왕복한 후 메모리

### 5.3 기대 효과

1차 목표:

- 초기 메모리 사용량 감소
- 페이지 전환 후 메모리 증가 폭 완화
- 긴 스트리밍 응답 중 브라우저 부담 완화

주의:

- dev 환경에서는 HMR과 sourcemap 때문에 production보다 크게 보일 수 있음
- 반드시 `npm run build` 결과와도 비교할 것

## 8. 이번 저장소 기준 결론

현재 코드만 보면 "치명적인 메모리 누수"가 바로 보이진 않는다.

하지만 아래 구조는 분명 개선 가치가 있다.

- 전체 페이지 eager import
- 대형 컴포넌트 내부의 정적 mock 데이터 집중
- 불필요한 reactive state
- 채팅의 글자 단위 스트리밍 업데이트

따라서 사내 반영 시에는:

- "누수 수정"보다는
- "초기 로드 범위 축소 + 렌더링 비용 절감 + 정적 데이터 정리"

방향으로 접근하는 것이 적절하다.
